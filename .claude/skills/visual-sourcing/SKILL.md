# 시각 자료 수집 (visual-sourcing)

도산 안창호 관련 사진·문서 이미지를 저작권 안전성이 확인된 것만 수집하고, 출처·연대·인물 메타데이터를 갖춰 사이트에 공급하는 방법을 정의한다. 역사 웹사이트의 신뢰는 이미지에서도 나온다 — 출처 불명 이미지 한 장이 사이트 전체의 사료적 신뢰를 무너뜨리고, 저작권 침해 이미지 한 장이 법적 문제를 만든다. 그래서 이 스킬은 "수집 → 판별 → 메타데이터 → 최적화"를 단일 절차로 묶고, 판별을 통과하지 못한 이미지는 아무리 좋아도 버린다.

## 1. 저작권 판별 기준

이미지마다 다음 순서로 판별하고, 판별 결과와 근거를 메타데이터에 기록하라.

### 판별 트리

1. **위키미디어 커먼즈에 라이선스 태그(PD, CC0, CC-BY 등)와 함께 등재되어 있는가?** → 태그와 근거 설명을 확인하고 그대로 인용. 단, 커먼즈도 오등재가 있으므로 태그의 근거(사망 연도, 공표 연도)가 아래 규칙과 모순되면 제외.
2. **한국 저작물:** 저작자 사망 후 70년 경과 시 퍼블릭 도메인. 1938년 이전에 사망한 작가·촬영자의 저작물은 안전. 단체명의 저작물·무명 저작물은 공표 후 70년. 안창호 본인(1938 순국)의 글·글씨는 퍼블릭 도메인이다.
3. **미국 저작물:** 1930년 이전(포함) 출판물은 퍼블릭 도메인(공표 후 95년 규칙, 2026년 기준). 미주 한인 신문(공립신보·신한민보)의 해당 시기 지면이 여기 속한다.
4. **기관 소장 자료(독립기념관·국사편찬위 등):** 원본이 퍼블릭 도메인이라도 기관이 스캔본에 이용조건을 붙이는 경우가 있다. 각 기관의 이용조건 페이지를 확인하고, "출처 표기 후 자유 이용" 이상이 아니면 제외.
5. **위 어디에도 확정적으로 해당하지 않으면 → 제외.**

### "불명확하면 제외"가 기본값인 이유

이 사이트는 공개 웹에 배포된다. 배포는 사적 이용이 아니라 **공중송신 행위**이므로, 판단이 틀렸을 때의 비용이 비대칭적이다 — 좋은 이미지를 하나 포기하는 비용은 작지만(대체 이미지를 찾거나 텍스트로 보완하면 된다), 침해 이미지를 배포한 비용(삭제 요구, 법적 책임, 사이트 전체의 신뢰 손상)은 크고 사후 회수가 어렵다. 한 번 배포된 사이트는 아카이빙·미러링되므로 "나중에 내리면 된다"도 성립하지 않는다. "아마 괜찮을 것"이라는 추정은 근거가 아니다 — 근거는 사망 연도·공표 연도·라이선스 문서 같은 확인 가능한 사실뿐이다.

촬영자·공표 연도를 특정할 수 없는 이미지는 아무리 희귀해도 수집하지 말고, 대신 `_workspace/03_content/images/excluded.md`에 다음 형식으로 기록해 후속 라운드에서 출처가 보강되면 재검토할 수 있게 하라.

```markdown
### 제외: 대성학교 교정 추정 사진
- 발견 위치: {URL}
- 제외 사유: 촬영자·공표 연도 미상, 소장처 이용조건 페이지 없음
- 재검토 조건: 독립기념관 소장 여부 확인 시 이용조건 재판별
```

제외 기록을 남기는 이유: 제외는 폐기가 아니라 보류이고, 기록이 없으면 다음 라운드가 같은 이미지를 같은 비용으로 다시 판별하게 된다.

## 2. 수집처 우선순위

| 순위 | 수집처 | 비고 |
|------|--------|------|
| 1 | 위키미디어 커먼즈 (commons.wikimedia.org) | 라이선스 태그 명시, 가장 안전 |
| 2 | 독립기념관 한국독립운동정보시스템 (search.i815.or.kr) | 이용조건 개별 확인 필수 |
| 3 | 국사편찬위 한국사데이터베이스 (db.history.go.kr) | 이용조건 개별 확인 필수 |
| 4 | 공훈전자사료관 (e-gonghun.mpva.go.kr) | 공공누리 유형 확인 |
| 5 | USC Korean American Digital Archive | 미주 시기 사진, rights 필드 확인 |
| 6 | 흥사단·도산안창호기념사업회 공개 자료 | 이용 허락 범위 확인 |

검색은 한글("안창호", "도산")·한자("安昌浩", "島山")·영문("Ahn Chang-ho", "An Ch'angho", "Dosan") 표기를 병행하라 — 아카이브마다 색인 언어가 다르므로 단일 표기 검색은 수집 누락을 만든다.

페이지 명세의 이미지 슬롯 수보다 **2배수**의 후보를 수집하라. 디자이너와 개발자가 톤·해상도·구도를 보고 고를 수 있어야 하며, 후보가 슬롯 수와 같으면 선택이 아니라 강제가 된다.

## 3. 이미지 메타데이터 스키마

모든 수집 이미지는 `_workspace/03_content/images/manifest.json`에 다음 레코드로 등록하라.

```json
{
  "images": [
    {
      "file": "1913_heungsadan_founding.webp",
      "original_file": "1913_heungsadan_founding_original.jpg",
      "title": "흥사단 창립 단우들",
      "date": "1913",
      "date_precision": "year",
      "depicted": ["안창호", "송종익"],
      "place": "로스앤젤레스",
      "source_org": "위키미디어 커먼즈",
      "source_url": "https://commons.wikimedia.org/wiki/File:...",
      "license": "Public Domain (저작자 사망 후 70년 경과)",
      "license_evidence": "촬영 1913년, 한국·미국 모두 PD 요건 충족",
      "caption": "1913년 로스앤젤레스에서 창립한 흥사단의 초기 단우들.",
      "credit": "출처: 위키미디어 커먼즈",
      "slot": "heungsadan-page-hero"
    }
  ]
}
```

필수 필드: `file, title, date, source_org, source_url, license, license_evidence, caption, credit`. 인물 사진이면 `depicted`도 필수 — 식별 불가 인물은 빈 배열이 아니라 `["미상"]`으로 명시해 "조사했으나 알 수 없음"과 "조사 안 함"을 구분하라. `license_evidence`를 별도 필드로 두는 이유: 라이선스 명칭만 적으면 나중에 검증할 수 없고, 판별 근거(사망/공표 연도)가 있어야 content-qa가 재검증할 수 있다.

## 4. 다운로드·리네이밍·최적화 절차

1. **다운로드:** 가능한 최고 해상도 원본을 받아라. 축소는 나중에 할 수 있지만 확대는 불가능하다. 다운로드 직후 파일이 실제 이미지인지(0바이트·HTML 에러 페이지가 아닌지) 확인하라.
2. **원본 보존:** 원본은 `site/assets/images/originals/`에 손대지 않고 보관. 최적화 실패·재처리 시의 복구 지점이고, 사료의 원본성 증빙이기도 하다.
3. **리네이밍:** `{연도}_{주제}_{식별어}.{ext}` 형식 (예: `1908_daesung_school.jpg`). 연도 미상은 `nd_` 접두. 한글·공백 파일명 금지 — URL 인코딩 문제와 끊어진 이미지를 만든다.
4. **최적화:** 표시용은 WebP로 변환하고 표시 크기에 맞춰 리사이즈(본문 폭 기준 최대 1600px, 썸네일 480px). 역사 사진은 과압축 시 입자가 뭉개져 사료 가치가 손상되므로 품질 80 이상을 유지하고, 변환 후 육안 확인하라.

```bash
# cwebp가 있으면
cwebp -q 82 -resize 1600 0 originals/1908_daesung_school.jpg -o 1908_daesung_school.webp
```

```python
# 없으면 Pillow로 (pip 불가 환경이면 원본 포맷 유지 + 리사이즈만이라도 수행)
from PIL import Image
img = Image.open("originals/1908_daesung_school.jpg")
img.thumbnail((1600, 1600))
img.save("1908_daesung_school.webp", "WEBP", quality=82)
```

5. **매니페스트 갱신:** 파일을 옮기거나 리네이밍한 즉시 manifest.json을 갱신하라. 파일 시스템과 매니페스트의 불일치는 frontend-developer의 끊어진 이미지로 직결된다.

### 절차 요약 (이미지 1장의 수명 주기)

```
발견 → 저작권 판별(1절) ─ 불통과 → excluded.md 기록
                       └ 통과 → 원본 다운로드 → originals/ 보존
                                → 리네이밍 → WebP 최적화 → 육안 확인
                                → manifest.json 레코드 작성 (메타데이터 전 필드)
```

어느 단계든 건너뛴 이미지는 미완성이다 — 특히 "일단 넣고 메타데이터는 나중에"는 금지. 나중은 오지 않고, 메타데이터 없는 이미지는 content-qa에서 출처 불명 결함이 된다.

## 산출물 검증

산출물 제출 전 다음을 직접 확인하라.

1. `manifest.json`이 유효한 JSON인지 파싱으로 확인 (`python3 -m json.tool`).
2. 매니페스트의 모든 `file`이 `site/assets/images/`에 실제 존재하는지, 역으로 디렉토리의 모든 이미지가 매니페스트에 등록되어 있는지 양방향 대조.
3. 모든 레코드에 필수 필드가 채워져 있는지 — 특히 `license_evidence`가 "확인 필요" 같은 미완 상태로 남아 있으면 미완성이다.
4. `excluded.md`에 제외 이미지와 사유가 기록되어 있는지 — 제외 기록이 0건인데 수집 후보 전원이 통과했다면, 판별을 실제로 수행했는지 스스로 의심하라.
5. 슬롯 명세 대비 후보 수가 2배수에 미달하는 슬롯 목록을 보고서에 명시.
