# 도산 디자인 시스템 (dosan-design-system)

도산 안창호 사이트의 시각 언어를 정의한다. 디자인 컨셉은 도산의 정신에서 직접 도출한다 — **무실역행(務實力行)의 절제**: 꾸밈없이 실질을 다하고(務實), 묵묵히 실행한다(力行). 이를 시각으로 번역하면 넉넉한 여백, 정직한 타이포그래피, 장식의 최소화다. 화려한 효과는 도산을 말하는 것이 아니라 도산을 가린다. 이 스킬이 모든 색·글꼴·컴포넌트의 단일 기준이며, 여기서 벗어난 임의 스타일은 팀의 시각적 일관성을 깨는 계약 위반이다.

## 글로벌 frontend-design 스킬과의 관계

이 스킬은 글로벌 `frontend-design` 스킬과 충돌하지 않는다. 역할 분담은 다음과 같다.

- **frontend-design:** 일반 품질 원칙 — 생성형 AI 특유의 범용 디자인 회피, 완성도, 창의적 폴리시. 이 원칙들은 그대로 따른다.
- **본 스킬:** 이 프로젝트 고유의 결정 — 컨셉, 토큰 값, 컴포넌트 스펙. frontend-design이 "좋은 디자인을 하라"를 말한다면, 본 스킬은 "이 사이트에서 좋은 디자인이 구체적으로 무엇인가"를 고정한다.

두 스킬의 지시가 충돌하는 것처럼 보이면 프로젝트 고유 토큰·컨셉(본 스킬)이 우선한다.

## 1. 디자인 컨셉 — 무실역행의 절제

- **여백이 위계다.** 구분선·박스·그림자 대신 여백으로 위계를 만든다. 역사 텍스트는 읽는 데 집중할 수 있어야 하고, 시각적 소음은 독해를 방해한다.
- **정직한 타이포.** 텍스트가 주인공이다. 본문은 세리프로 길게 읽기 좋게, 장식적 디스플레이 폰트는 쓰지 않는다.
- **장식 최소.** 그라디언트, 글로우, 과한 애니메이션 금지. 움직임은 의미(시간의 흐름, 경로)를 전달할 때만 허용한다.
- **재료의 은유.** 한지(배경), 먹(텍스트), 단청 빨강(강조 1색), 청자색(보조)만 사용한다. 단청 빨강은 인장(印章)처럼 — 한 화면에 한두 번, 가장 중요한 곳에만 찍는다. 강조색이 흔해지면 강조가 아니다.

## 2. 컬러 토큰

`site/css/tokens.css`에 다음 커스텀 프로퍼티를 정의하고, 모든 색은 토큰 참조로만 사용하라. 하드코딩된 hex가 CSS에 흩어지면 톤 조정이 전수 수정이 된다.

```css
:root {
  /* 한지 — 배경 */
  --paper:        #F7F3EB;  /* 기본 배경 */
  --paper-dim:    #EFE8DB;  /* 카드·인용 블록 배경 */

  /* 먹 — 텍스트 */
  --ink:          #1A1815;  /* 본문 텍스트 */
  --ink-soft:     #4A453E;  /* 보조 텍스트(캡션·메타) */

  /* 단청 — 강조 (인장처럼 아껴 쓸 것) */
  --dancheong:    #B5342B;  /* 링크·핵심 강조·현재 위치 */

  /* 청자 — 보조 */
  --celadon:      #6E8B74;  /* 대형 제목 보조·테두리·비텍스트 UI */

  /* 시맨틱 별칭 — 컴포넌트는 별칭만 참조 */
  --bg:           var(--paper);
  --bg-raised:    var(--paper-dim);
  --text:         var(--ink);
  --text-muted:   var(--ink-soft);
  --accent:       var(--dancheong);
  --on-accent:    #FFFFFF;  /* 단청 배경 위 텍스트 */
}
```

### WCAG AA 대비표 (사용 전 반드시 확인)

| 전경 / 배경 | 대비율 | 일반 텍스트 AA (≥4.5) | 큰 텍스트·UI AA (≥3.0) | 용도 |
|---|---|---|---|---|
| `--ink` / `--paper` | 16.0:1 | 통과 (AAA) | 통과 | 본문 기본 |
| `--ink` / `--paper-dim` | 14.5:1 | 통과 (AAA) | 통과 | 카드 본문 |
| `--ink-soft` / `--paper` | 8.6:1 | 통과 (AAA) | 통과 | 캡션·메타 |
| `--dancheong` / `--paper` | 5.4:1 | 통과 | 통과 | 링크·강조 텍스트 |
| `--dancheong` / `--paper-dim` | 4.9:1 | 통과 | 통과 | 카드 내 링크 |
| `#FFFFFF` / `--dancheong` | 6.0:1 | 통과 | 통과 | 강조 버튼 |
| `--celadon` / `--paper` | 3.4:1 | **불통과** | 통과 | 큰 제목(24px+)·테두리·아이콘만 |
| `--dancheong` / `--ink` | 2.9:1 | **불통과** | **불통과** | **금지 조합** |

- `--celadon`은 본문 크기 텍스트에 절대 쓰지 마라 — 18.66px(14pt) bold 또는 24px 이상의 큰 텍스트, 비텍스트 UI 요소만 허용.
- 새 색 조합을 도입하면 표에 추가하기 전 대비율을 실측하라(브라우저 DevTools 또는 WebAIM Contrast Checker 공식). 표에 없는 조합은 미검증 조합이다.

## 3. 타이포그래피

```css
:root {
  --font-serif: "Noto Serif KR", "Nanum Myeongjo", serif;       /* 제목·본문 */
  --font-sans:  "Pretendard", "Apple SD Gothic Neo", sans-serif; /* UI·메타·캡션 */

  /* 크기 스케일 (1.25배) — 임의 px 금지, 스케일에서만 선택 */
  --fs-sm:   0.8rem;    /* 캡션·크레딧 */
  --fs-base: 1rem;      /* 본문 (root 18px 권장) */
  --fs-md:   1.25rem;   /* 소제목 h4 */
  --fs-lg:   1.563rem;  /* h3 */
  --fs-xl:   1.953rem;  /* h2 */
  --fs-2xl:  2.441rem;  /* h1 */

  --lh-body: 1.8;       /* 한글 장문 가독 행간 */
  --lh-tight: 1.3;      /* 제목 */
  --measure: 42rem;     /* 본문 최대 폭 — 한 줄 35~45자 */
}
```

- 본문은 세리프, UI 요소(내비·버튼·필터)는 산세리프 — 읽는 것과 조작하는 것을 글꼴로 구분한다.
- 웹폰트는 Google Fonts 등 안정적 CDN에서 `display=swap`으로 로드하고 시스템 폰트 폴백을 항상 유지하라. 폰트 로드 실패가 빈 텍스트가 되어서는 안 된다.
- 한자 병기는 본문과 같은 글꼴·크기로, 괄호 안에. 한자만 작게 줄이는 스타일은 가독성을 해친다.

## 4. 컴포넌트 스펙

- **연표 카드:** `--bg-raised` 배경, 좌측에 날짜(산세리프 `--fs-sm`), 제목(세리프 `--fs-md`), 요약 1~2줄. `disputed: true` 사건은 단청색 점선 테두리 + "기록 상충" 라벨 — 색만으로 구분하지 마라(색각 이상 대응).
- **인용 블록:** 사료 원문 인용은 `--bg-raised` 배경, 좌측 3px `--celadon` 보더, 세리프 이탤릭 금지(한글 이탤릭은 가짜 기울임). 출처 크레딧은 블록 하단 `--text-muted`.
- **사료 뷰어:** 원문 이미지 + 전사 텍스트 + 현대어 풀이의 3단 구성. 각 단의 라벨을 명시해 무엇이 원문이고 무엇이 해석인지 시각적으로 구분.
- **인물 카드:** 인물 사진(없으면 이니셜 플레이스홀더), 이름(한자 병기), 생몰년, 도산과의 관계 유형 라벨, 한 줄 요약. 관계 유형은 텍스트 라벨로 — 색상 코딩만으로 표현 금지.

## 5. 사진 처리

- 시대 사진은 **흑백(grayscale) 통일 톤**으로 처리해 출처가 제각각인 이미지들의 시각적 일관성을 만든다. 원본이 세피아·변색이어도 동일 필터를 적용하라.
- 처리는 CSS `filter: grayscale(1)`로 — 원본 파일을 변형하지 않으므로 사료 원본성이 보존되고 정책 변경 시 한 줄 수정으로 끝난다.
- 모든 사진에 캡션(설명 + 연대)과 크레딧(출처)을 `<figcaption>`으로 표기 — manifest.json의 caption/credit 필드를 그대로 사용하고 임의 작성하지 마라.
- 모든 `<img>`에 의미 있는 alt 텍스트 필수. 장식 이미지는 이 사이트에 존재하지 않는다 — 모든 이미지가 사료다.

## 6. 반응형 브레이크포인트

```css
/* 모바일 우선 — 기본 스타일은 모바일 */
@media (min-width: 640px)  { /* 태블릿: 2단 카드 그리드 */ }
@media (min-width: 1024px) { /* 데스크톱: 본문 + 사이드 내비 */ }
```

- 모바일 우선으로 작성하라 — 기본 CSS가 가장 제약이 큰 환경에서 동작하면 확장은 추가일 뿐이지만, 데스크톱 우선은 축소 과정에서 깨진다.
- 본문 폭은 모든 뷰포트에서 `--measure`를 넘지 않는다. 넓은 화면에서 본문이 늘어나는 것은 가독성 결함이다.
- 연표·지도 같은 인터랙티브 컴포넌트는 3개 뷰포트(360px/768px/1280px)에서 각각 동작 확인 — interactive-viz 스킬의 폴백 규칙과 함께 적용하라.

## 산출물 검증

1. **토큰 단일성:** `site/css/` 전체에서 `#[0-9A-Fa-f]{3,6}` 하드코딩 hex를 grep — tokens.css 외부에서 발견되면 토큰 참조로 교체하라.
2. **대비 검증:** 실제 사용 중인 모든 전경/배경 조합이 2절 대비표에 있고 용도에 맞는 등급을 통과하는지 확인. 표에 없는 조합 발견 시 실측 후 표를 갱신하거나 제거.
3. **금지 조합 검사:** `--dancheong` 텍스트가 `--ink` 계열 배경 위에, `--celadon`이 본문 크기 텍스트에 쓰인 곳이 없는지 검사.
4. **컨셉 정합:** 화면당 단청색 사용 빈도를 확인 — 한 화면에 강조색이 3곳 이상이면 절제 원칙 위반을 의심하라.
5. **design-system.md와 tokens.css 동기화:** 문서의 토큰 값과 CSS 실제 값이 일치하는지 대조 — 문서와 코드의 불일치는 후속 작업자를 오도한다.
