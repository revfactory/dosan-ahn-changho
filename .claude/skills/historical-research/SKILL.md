# 역사 조사 방법론

도산 안창호 일대기 조사의 모든 산출물은 후속 검증(Phase 2)과 콘텐츠 작성(Phase 3)의 원료가 된다. 원료 단계에서 출처가 누락되거나 형식이 제각각이면 하류 전체가 오염되므로, 모든 연구원이 동일한 출처 위계·스키마·표기 규칙으로 수집해야 한다. 이 스킬은 "무엇을 믿을 것인가(출처 위계)", "어디서 찾을 것인가(아카이브)", "어떻게 기록할 것인가(스키마)"를 규정한다.

## 1. 출처 위계 5단계

출처의 가치는 동등하지 않다. 같은 사실이라도 어느 출처에서 왔는지에 따라 후속 신뢰도 등급(A~D)이 달라지므로, 수집 시점에 출처 유형을 정확히 기록하라.

| 단계 | 유형 | 예시 | 비고 |
|------|------|------|------|
| 1 | 1차 사료 | 도산 일기, 서한, 연설문 원문, 동시대 신문(공립신보·신한민보·독립신문), 일제 신문조서·판결문 | 가장 강한 근거. 단, 작성자 의도 비판 필요(S03 참조) |
| 2 | 학술 논문·연구서 | 학술지 논문, 대학·연구기관 출판 전기 | 사료 비판을 거친 2차 해석 |
| 3 | 공공기관 DB | 한국사데이터베이스, 공훈전자사료관, 독립기념관 정보시스템 | 기관 검수를 거친 정리 자료 |
| 4 | 백과사전 | 한국민족문화대백과, 위키백과 | 출발점으로만 사용 — 반드시 상위 출처로 역추적 |
| 5 | 일반 웹 | 블로그, 언론 칼럼, 단체 홍보물 | 단독 근거 불가. 단서 발견용으로만 |

4·5단계에서 발견한 사실은 그대로 기록하지 말고 1~3단계 출처로 거슬러 올라가 확인하라. 백과사전의 오류가 웹 전체에 복제되는 일이 흔하므로, 같은 문장이 여러 사이트에 반복된다는 것은 독립 확인이 아니다.

## 3. 사건 레코드 JSON 스키마 (공통)

모든 연구원의 사건 산출물은 이 스키마 하나로 통일한다. 스키마가 갈라지면 Phase 2의 timeline 병합이 불가능해진다. 이 스키마는 timeline-construction(S05)에서도 그대로 참조한다 — 재정의하지 마라.

```json
{
  "id": "evt-{주제약어}-{일련번호}",        // 예: evt-shinminhoe-003. 전역 유일해야 함
  "title": "사건 제목 (간결한 명사형)",
  "date": {
    "start": "YYYY-MM-DD",                  // 알 수 없는 자리는 precision으로 표현
    "end": "YYYY-MM-DD",                    // 단일 시점이면 start와 동일 또는 null
    "precision": "day|month|year|range",    // 사료가 보장하는 정밀도. 과장 금지
    "calendar": "solar|lunar"               // 원본 사료의 역법. 1896 이전은 음력 가능성 높음(S05 참조)
  },
  "place": {
    "name": "당대 지명",                     // 예: "리버사이드 파차파 캠프"
    "modern_name": "현대 지명",              // 예: "미국 캘리포니아주 리버사이드"
    "lat": 0.0,                             // 좌표 미상이면 null — 임의 추정 금지
    "lng": 0.0
  },
  "actors": ["인물명"],                      // 한글 표준 표기. 이명은 network 정규화 테이블에서 처리
  "orgs": ["단체명"],
  "summary": "1~2문장 요약",
  "detail": "상세 서술. 사실만 — 해석은 report.md로",
  "sources": [
    {
      "type": "primary|academic|institutional|encyclopedia|web",  // 출처 위계 1~5단계 대응
      "title": "출처 제목",
      "locator": "권·호·면 또는 문서번호",
      "url": "접근 URL (없으면 null + 소장처 기록)"
    }
  ],
  "confidence": "A|B|C|D",                  // 잠정 등급. 최종 등급은 fact-checker가 확정
  "tags": ["교육", "결사", "언론", "사상", "가족", "이동"]
}
```

날짜·장소·출처 3요소 중 하나라도 비면 `confidence`를 `low`(D)로 내려 기록하라. 빈 필드를 그럴듯한 값으로 채우는 것이 가장 위험한 오류다 — 후속 검증자는 채워진 값을 의심하지 않기 때문이다.

## 4. 검색 전략

같은 대상이 출처마다 다른 표기로 등장하므로, 단일 표기 검색은 체계적 누락을 만든다.

1. **표기 병행:** 한글("안창호") + 한자("安昌浩") + 호("도산", "島山") + 영문("Ahn Chang-ho", "An Ch'angho", "Tosan/Dosan")을 모두 시도하라. 단체도 동일 — "공립협회"와 "Kongnip Hyŏphoe", "Mutual Assistance Society".
2. **순서:** WebSearch로 소재를 파악 → 유망한 결과는 WebFetch로 원문 페이지를 직접 읽어 확인. 검색 결과 스니펫만 보고 인용하지 마라 — 스니펫은 문맥이 잘려 의미가 뒤집히는 경우가 있다.
3. **연쇄 추적:** 학술 논문의 각주에서 1차 사료의 소재(소장처·문서번호)를 수확하라. 논문 본문보다 각주가 더 값진 경우가 많다.
4. **실패 기록:** 검색했으나 찾지 못한 것도 report.md에 "탐색했으나 미발견"으로 기록하라. 다음 라운드 연구원이 같은 탐색을 반복하지 않게 하는 것이 공백 맵(gaps.md)의 재료다.
5. **도구 실패:** WebFetch가 실패하면 1회 재시도 후, 그래도 실패하면 URL과 실패 사실을 sources에 남기고 진행하라. 막힌 출처 하나에 매달려 라운드를 지연시키지 마라.

## 5. 미확인·추정 표기 규칙

사료가 침묵하는 구간을 서사로 메우고 싶은 유혹이 가장 큰 위험이다. 공백을 공백으로 남기는 것이 조사의 정직성이며, 후속 단계가 공백을 인지해야 보완 조사를 지시할 수 있다.

- **미확인 사실:** 레코드를 만들되 `confidence: "D"` + summary에 "〔미확인〕" 접두. 삭제하지 마라 — 미확인 주장의 존재 자체가 정보다.
- **추정:** 학술 출처가 추정한 것만 기록 가능. "~로 추정된다(출처)" 형식으로, 연구원 자신의 추정은 events.json에 넣지 말고 report.md의 "가설" 절에 분리하라.
- **상충:** 출처 간 진술이 다르면 양쪽 모두 sources에 넣고 detail에 상충 내용을 병기하라. 어느 쪽도 삭제하지 마라 — 채택 판단은 cross-validator의 몫이다.
- **범위 날짜:** "1905년 봄" 같은 표현은 `start: 1905-03-01, end: 1905-05-31, precision: "range"`로. 임의로 중간값을 찍지 마라.

## 6. 산출물 형식

각 연구원은 두 파일을 산출한다.

- `_workspace/01_research/{agent}_events.json` — 위 스키마의 사건 레코드 배열. 유효한 JSON이어야 한다.
- `_workspace/01_research/{agent}_report.md` — 조사 경과·핵심 발견·미발견 목록·가설·다른 연구원에 넘길 인계 사항. 사건 데이터를 본문에 중복 서술하지 말고 `evt-` id로 참조하라.

network-researcher는 events.json 대신 `network_nodes_edges.json`(S06 스키마), primary-source-researcher는 `primary-source_catalog.json`(S03 스키마)을 산출하되, 보고서 형식은 동일하다.

### 보완 라운드 (재조사 시)

research-director의 공백 맵(gaps.md)이나 사용자의 "조사 다시/보완" 요청으로 재호출되면, 기존 산출물을 먼저 읽고 **증분으로** 작업하라 — 기존 레코드를 버리고 처음부터 다시 만들면 이미 검증에 들어간 id가 끊어진다. 새 사건은 새 id로 추가하고, 기존 사건의 수정은 id를 유지한 채 필드만 갱신하며, 수정 내역을 report.md에 "변경 이력" 절로 남겨라.

## 산출물 검증

제출 전 다음을 스스로 확인하라. 하나라도 실패하면 제출하지 말고 수정하라.

1. `python3 -c "import json; json.load(open('파일'))"` 으로 JSON 파싱 성공 확인.
2. 모든 레코드에 `sources`가 1개 이상 있고, 각 source에 `type`과 `title`이 있는가. 출처 누락률이 10%를 넘으면 research-director가 반려한다.
3. `id` 전역 유일성 — 중복 id는 timeline 병합 시 데이터 유실을 일으킨다.
4. `precision`이 사료가 실제 보장하는 수준과 일치하는가 — 출처에 연도만 있는데 `day`로 적지 않았는가.
5. report.md에 "미발견 목록" 절이 존재하는가 — 공백 0건이라면 그렇게 명시했는가.
