# 관계망 데이터 구축

도산의 생애는 혼자가 아니라 관계 속에서 전개되었다 — 독립협회에서 임시정부까지 모든 활동이 인물·조직 네트워크 위에서 이루어졌다. 관계망 데이터의 목적은 이 구조를 시각화 가능한 그래프로 만드는 것이지만, 그래프의 함정은 "선 하나 긋기"가 너무 쉽다는 데 있다. 근거 없는 엣지는 그럴듯한 시각화를 만들고, 그럴듯한 시각화는 거짓을 사실처럼 보이게 한다. 따라서 이 스킬의 핵심 원칙은 모든 관계를 검증 가능한 사건에 정박시키는 것이다.

## 1. nodes/edges 스키마

산출물(`network_nodes_edges.json` → 확정본 `_workspace/02_verified/network.json`)은 다음 구조를 가진다.

```json
{
  "nodes": [
    {
      "id": "per-{일련번호} 또는 org-{일련번호}",  // 인물은 per-, 조직은 org- 접두로 구분
      "name": "표준 한글 표기",                    // 정규화 테이블의 대표 표기 (3절)
      "hanja": "한자 표기",                        // 인물·단체의 한자명. 없으면 null
      "alt_names": ["이명", "호", "영문명"],        // 사료에 등장하는 모든 변형 표기 — 검색·대조용
      "birth": "YYYY 또는 YYYY-MM-DD",            // 인물 노드만. 조직은 창립일을 사용
      "death": "YYYY 또는 YYYY-MM-DD",            // 인물 노드만. 조직은 해산일. 미상이면 null
      "role": "한 줄 정체성",                      // 예: "독립운동가·임시정부 국무령"
      "summary": "2~3문장 — 도산과의 접점 중심"
    }
  ],
  "edges": [
    {
      "source": "노드 id",                        // 반드시 nodes에 존재하는 id
      "target": "노드 id",
      "type": "comrade|conflict|mentor|family|membership|patron",  // 2절 분류
      "period": {
        "from": "YYYY",                           // 관계 성립 시기. 불명이면 null + description에 사유
        "to": "YYYY"                              // 관계 종료(사망·결별·단체 해산). 지속이면 null
      },
      "evidence_event_ids": ["evt-..."],          // 근거 사건 레코드 id — 빈 배열 불가 (5절)
      "description": "관계의 성격 1~2문장 — 시기 변화·복합 관계 설명 포함"
    }
  ]
}
```

- 인물-조직 관계는 `membership` 엣지로, 조직-조직 관계(전신·후신·산하)도 엣지로 표현하라.
- 한 쌍 사이에 성격이 다른 관계가 시기별로 존재하면(예: 동지였다가 노선 갈등) 엣지를 분리하고 각각 `period`를 부여하라. 하나의 엣지에 뭉뚱그리면 시기 슬라이더 시각화에서 거짓 관계가 표시된다.

## 2. 관계 유형 분류

| type | 정의 | 판별 기준 |
|------|------|----------|
| `comrade` (동지) | 같은 운동·조직에서 협력한 관계 | 공동 활동 사건이 근거로 존재 |
| `conflict` (갈등) | 노선·정책·개인적 대립 관계 | 대립이 기록된 사건이 근거로 존재 |
| `mentor` (사제) | 가르침·지도 관계 (양방향 모두) | 교육 기관·지도 활동 사건이 근거 |
| `family` (가족) | 혼인·혈연 관계 | 가족 관계 기록이 근거 |
| `membership` (조직소속) | 인물의 조직 가입·직책 | 가입·직책 임명 사건이 근거 |
| `patron` (후원) | 재정·물질 지원 관계 | 후원 기록이 근거 |

분류 시 유의:

- **갈등 관계를 동등하게 수집하라.** 이승만과의 임정 노선 갈등, 이광수와의 민족개조론 차이 같은 대립은 도산 네트워크의 핵심 구조다. 갈등을 누락하면 관계망이 미화된 초상이 되고, 균형을 잃은 데이터는 신뢰를 잃는다 (S07 어조 원칙과 같은 이유).
- **복합 관계는 분리하라.** 안창호-이광수는 동지(`comrade`, 수양동우회)이면서 사상적 긴장(`conflict`, 민족개조론) 관계다. 두 엣지로 분리하고 각각 근거를 달아라.
- **유형이 애매하면 근거 사건이 보여주는 것만 분류하라.** "아마 친했을 것"으로 `comrade`를 부여하지 마라 — 근거 사건이 공동 활동이면 `comrade`, 그 이상은 부여하지 않는다.

## 3. 인명 정규화

같은 인물이 본명·호·이명·창씨명·영문 표기로 사료마다 다르게 등장한다. 정규화하지 않으면 한 인물이 그래프에 여러 노드로 흩어져 관계망이 파편화된다.

1. **대표 표기 1개를 정하라.** 원칙: 학계 통용 한글 본명(예: "안창호", "이승만"). 호가 더 통용되는 경우라도 노드 `name`은 본명으로, 호는 `alt_names`에.
2. **모든 변형을 `alt_names`에 수집하라.** 호(도산), 한자(安昌浩), 영문(Ahn Chang-ho, An Ch'angho), 가명·이명 전부. 이 배열이 풍부할수록 다른 산출물과의 자동 대조(같은 인물 탐지)가 정확해진다.
3. **정규화 테이블을 별도 산출하라.** relation-analyst는 `변형 표기 → 노드 id` 매핑 테이블을 만들어, 다른 산출물(events.json의 actors, pages의 인명)이 같은 기준으로 통일되도록 공유하라.
4. **동명이인을 경계하라.** 표기가 같다고 자동 병합하지 마라 — 생몰·활동 시기·소속이 일치할 때만 같은 인물로 판정하고, 판정 근거를 노드 summary에 남겨라.

### 조직 노드의 변천 처리

도산이 관여한 조직은 개편·통합·개칭이 잦다(공립협회→대한인국민회, 수양동맹회·동우구락부→수양동우회). 전신과 후신을 하나의 노드로 합치면 시기별 관계가 왜곡되고, 완전히 분리하면 연속성이 사라진다. 원칙:

- 법적·조직적 단절이 있으면 별도 노드로 만들고, 전신-후신 관계를 `membership`이 아닌 별도 설명의 엣지(`description`에 "전신/후신" 명시)로 연결하라.
- 단순 개칭(동일 조직의 이름 변경)은 한 노드로 유지하고 옛 이름을 `alt_names`에 넣어라.
- 판정이 애매하면 학술 연구의 통설을 따르고 판정 근거를 `summary`에 기록하라 — 판정 자체도 검증 대상 주장이다.

## 4. 수집·정규화 워크플로우

1. **핵심 인물·조직 시드:** 청사진에 명시된 인물(이승만·김구·이광수·서재필·양기탁·신채호·이동휘·조만식·송종익·이혜련 등)과 조직(독립협회·공립협회·신민회·대한인국민회·흥사단·임시정부·수양동우회)을 시드 노드로 만들어라.
2. **타 연구원 산출물 크롤링:** 다른 연구원의 events.json에서 `actors`·`orgs` 필드를 전수 추출해 시드에 없는 인물·조직을 발견하라. 관계망 연구가 단독 검색에만 의존하면 시기별 연구원이 발굴한 주변 인물이 통째로 누락된다.
3. **노드 생성과 정규화:** 추출된 표기를 3절 규칙으로 정규화해 노드를 만들어라. 이때 생몰·역할이 미상인 인물도 노드로 만들되 `summary`에 미상임을 명시하라 — 보완 라운드의 조사 대상 목록이 된다.
4. **엣지 생성:** 근거 사건이 있는 관계만 엣지로 만들어라(5절). 사건에 두 인물이 공동 등장한다는 것만으로는 관계가 아니다 — 사건 서술이 상호작용을 보여줄 때만 엣지를 만들고, 공동 등장은 후보로만 기록하라.
5. **시기 속성 부여:** 각 엣지의 `period`를 근거 사건들의 날짜 범위에서 도출하라. 근거 사건이 1909년 하나뿐인데 `from: 1907`로 적으면 그 2년은 출처 없는 주장이다 — 확장하려면 근거 사건을 추가하라.
6. **교차 검증 인계:** relation-analyst는 확정 전에 timeline.json과의 id 교차 참조 무결성을 검사하고, 근거 미달 엣지를 분리하라.

## 5. 근거 사건 참조 의무

**모든 엣지는 `evidence_event_ids`에 근거 사건 id를 1개 이상 가져야 한다.** "친분이 있었다"는 단독 서술의 엣지는 금지다.

왜 이 의무가 절대적인가: 근거 사건이 없는 관계는 검증 경로가 없다. fact-checker가 등급을 매길 수 없고, content-qa가 대조할 원본이 없으며, 사이트 방문자가 "이 선의 근거가 무엇인가"를 물었을 때 답할 수 없다. 관계망에서 근거 없는 엣지 하나는 전체 그래프의 신뢰를 무너뜨린다 — 어느 선이 근거 있는 선인지 구별할 수 없게 되기 때문이다.

운영 규칙:

- 근거 사건이 아직 events.json에 없으면, 엣지를 만들기 전에 해당 사건 레코드를 먼저 만들거나 담당 연구원에게 수집을 요청하라. 순서는 항상 사건 → 엣지다.
- 검증 단계(relation-analyst)에서 근거 없는 엣지를 발견하면 삭제하지 말고 D등급으로 분리 보관하라(`network.json` 본체에서 제외하고 별도 `network_unverified.json`에). 삭제는 정보 소실이고, 본체 포함은 오염이다 — 분리가 둘 다 피한다.
- `evidence_event_ids`의 모든 id는 timeline.json의 사건 id와 교차 참조 무결성을 유지해야 한다. 참조가 끊긴 엣지는 근거 없는 엣지와 같다.
- 엣지의 `period`는 근거 사건의 날짜와 모순되면 안 된다 — 관계 시작 연도보다 이른 근거 사건이 달려 있으면 둘 중 하나가 틀린 것이다.

## 재구축 시 (관계망 다시/업데이트)

기존 network.json이 있으면 노드·엣지 id를 보존한 채 증분 갱신하라 — 노드 id는 시각화 코드와 다른 산출물이 참조할 수 있는 키다. 새 인물·관계는 추가로, 잘못된 엣지는 삭제 대신 `network_unverified.json`으로 이동으로 처리하고, 변경 내역을 report에 남겨라.

## 산출물 검증

확정 전 다음을 스크립트로 확인하라 (눈 검사로는 참조 무결성을 보장할 수 없다).

1. JSON 파싱 성공, `nodes`·`edges` 배열 존재.
2. 노드 id 전역 유일, 접두(`per-`/`org-`) 규칙 준수.
3. 모든 엣지의 `source`·`target`이 nodes에 실재 (끊어진 노드 참조 0건).
4. 모든 엣지의 `type`이 6개 허용값 중 하나.
5. 모든 엣지의 `evidence_event_ids`가 비어 있지 않고, 각 id가 timeline.json(또는 Phase 1 events.json)에 실재.
6. `alt_names` 간 충돌 — 서로 다른 노드가 같은 변형 표기를 공유하면 동명이인 판정 기록이 있는지 확인.
7. 갈등(`conflict`) 엣지가 0건이면 의심하라 — 도산의 생애에는 기록된 갈등이 분명히 존재하므로, 0건은 수집 누락의 신호다.
8. 고립 노드(엣지가 하나도 없는 노드)의 존재 사유 — 의도된 보완 대상인지, 엣지 수집 누락인지를 report에 분류해 기록했는가.
9. 엣지 `period`와 근거 사건 날짜의 모순 0건 — 근거 사건이 period 범위 밖에 있는 엣지를 스크립트로 추출해 확인하라.
10. 시드 인물·조직(청사진 명시 목록)이 전부 노드로 존재하는가 — 핵심 인물의 누락은 크롤링 이전 단계의 실패다.
