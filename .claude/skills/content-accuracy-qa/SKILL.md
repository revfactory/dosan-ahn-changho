# 콘텐츠 정확성 QA (content-accuracy-qa)

이 스킬은 "사이트가 검증된 사실만, 검증된 그대로 말하는가"를 판정하는 방법을 정의한다. Phase 2가 사실을 검증했고 Phase 3~4가 그것을 사이트로 옮겼다면, 남은 위험은 단 하나 — **옮기는 과정의 변형**이다. 오타가 된 연도, 뒤바뀐 인명, 윤색된 인용문, 검증 등급이 낮은 주장의 무단 승격. 역사 사이트에서 이런 오류는 코드 버그보다 치명적이다. 코드 버그는 화면이 깨져서 보이지만, 틀린 연도는 멀쩡해 보이는 채로 거짓을 가르친다. 그래서 이 QA의 원칙은 전수 대조이고, 판정 기준은 원본 일치다.

## 0. 전수 원칙 — 표본 검사가 무효인 이유

날짜·인명·지명·인용문은 **100% 전수 검사**한다. 표본 검사는 결함률을 추정하는 도구인데, 이 사이트의 허용 결함률은 0이다 — "95%의 날짜가 정확하다"는 결과는 행동 지침(어느 5%를 고칠지)을 주지 못하므로 QA로서 무의미하다. 전수가 가능하려면 손이 아니라 추출·대조의 기계화가 필요하다. 이 스킬의 절차가 그 기계화 방법이다.

## 1. 표시 텍스트 추출법

대조의 입력은 "사용자에게 실제로 보이는 텍스트"다. 소스가 아니라 표시 기준으로 추출하라.

1. **HTML 정적 텍스트:** `site/*.html`에서 태그 제거 후 텍스트 추출 (Python `html.parser`로 script/style 제외).
2. **데이터 주도 텍스트:** 콘텐츠가 fetch로 렌더되므로 HTML만 보면 빈 페이지다. `site/data/*.json`의 표시 대상 필드(title, summary, detail, caption, credit 등)를 함께 추출하라 — 어떤 필드가 화면에 표시되는지는 architecture.md의 계약과 소비 JS에서 확인한다.
3. **검사 대상 항목 추출:** 추출 텍스트에서 다음을 패턴으로 수집한다.
   - 날짜: `\d{4}년`, `\d{4}\.\d{1,2}`, "N월 N일" 등 + 음력 표기
   - 인명·지명·단체명: network.json의 노드 명단 + synthesis의 정규화 표기를 사전으로 매칭
   - 인용문: 따옴표·인용 블록 내 텍스트
   - 각주 마커: `[ref:id]` 또는 렌더된 각주 링크

추출 결과는 `(텍스트, 출현 위치)` 쌍으로 목록화하라 — 위치 없는 결함 보고는 수정자가 다시 찾아야 한다. 추출은 손이 아니라 스크립트로 — 손 추출은 전수성을 보장할 수 없다. 표준 라이브러리 기반 패턴 예시:

```python
# HTML 표시 텍스트 추출 — script/style 제외
from html.parser import HTMLParser

class TextExtractor(HTMLParser):
    SKIP = {"script", "style", "noscript"}
    def __init__(self):
        super().__init__(); self.texts = []; self._skip = 0
    def handle_starttag(self, tag, attrs):
        if tag in self.SKIP: self._skip += 1
    def handle_endtag(self, tag):
        if tag in self.SKIP: self._skip -= 1
    def handle_data(self, data):
        if not self._skip and data.strip():
            self.texts.append((data.strip(), self.getpos()))  # (텍스트, 행 위치)
```

```python
# JSON 표시 필드 추출 — 계약에 명시된 표시 필드만
import json, re
DISPLAY_FIELDS = {"title", "summary", "detail", "caption", "credit"}  # 계약에서 가져올 것
DATE_RE = re.compile(r"\d{4}년(?:\s*\d{1,2}월(?:\s*\d{1,2}일)?)?|\d{4}[.\-]\d{1,2}(?:[.\-]\d{1,2})?")

def walk(obj, path=""):
    if isinstance(obj, dict):
        for k, v in obj.items():
            if k in DISPLAY_FIELDS and isinstance(v, str):
                yield (f"{path}.{k}", v)
            yield from walk(v, f"{path}.{k}")
    elif isinstance(obj, list):
        for i, v in enumerate(obj):
            yield from walk(v, f"{path}[{i}]")
```

## 2. 전수 대조 절차

추출 항목을 원본 DB와 대조한다. 원본의 우선순위: `claims_register.json`(등급) → `timeline.json`(날짜·장소) → `network.json`(인명·관계) → `synthesis/*.md`(서술 맥락).

1. **날짜:** 사이트의 모든 날짜 표기를 timeline.json의 해당 사건과 대조. 검사 포인트 — 값 일치, precision 일치(데이터가 `year`인데 사이트가 월일까지 표시하면 결함: 데이터보다 정밀한 표시는 창작이다), 음양력 표기 유지.
2. **인명·지명:** 정규화 표기(한글+한자 병기 규칙)와 대조. 이명(異名)·호가 섞여 쓰였다면 style_rules.md의 표기 규칙 위반 여부 확인.
3. **인용문:** 원문 글자 단위 대조 — 인용은 요약이 아니므로 한 글자의 차이도 결함이다. 현대어 풀이가 병기 규칙대로 붙어 있는지 확인.
4. **수치·서술:** 본문 문단의 사실 주장을 claims_register의 해당 claim과 대조해 의미 변형(과장·단정화) 여부 확인.

**판정 규칙: 원본과 사이트가 다르면 무조건 결함이다.** "사소한 차이", "표현상 차이" 면제는 없다 — 면제 판단을 QA가 하기 시작하면 면제 기준이 사람마다 달라지고 전수 원칙이 무너진다. 원본이 틀렸다고 판단되면 사이트를 통과시키는 것이 아니라 원본 관리자(synthesis-editor/timeline-analyst)에게 회부하라. QA는 일치를 판정하고, 진실은 검증 DB가 판정한다.

## 3. 끊어진 각주 검사

각주는 사이트 신뢰의 마지막 연결고리다 — 끊어진 각주는 "출처 있음"을 가장한 "출처 없음"이다.

1. 본문(pages 데이터 + HTML)의 모든 출처 마커/각주 링크를 추출.
2. citations.json의 refs와 양방향 대조:
   - 마커는 있는데 매핑이 없음 → 결함 (citation-manager 회부)
   - 매핑은 있는데 본문에 마커 없음 → 고아 참조, 목록화
3. refs의 source_id가 sources에 실존하는지, references.html에 모든 source가 렌더되는지 확인.
4. 사실을 서술하는 문단에 마커가 하나도 없으면 무출처 문단으로 목록화 — 등급 미상의 주장이 숨어 있는 경로다.

## 4. C/D등급 노출 검사

synthesis 규칙상 본문은 A·B등급 주장으로만 구성되고, C등급은 "~로 전해진다" 같은 한정 표지와 함께만, D등급은 노출 금지다. 이 규칙이 사이트까지 살아남았는지 검사하라.

1. claims_register.json에서 C·D등급 claim 목록을 추출.
2. 각 claim의 핵심 표현(고유 인명·날짜·사건 조합)으로 사이트 전체 텍스트를 검색.
3. D등급 발견 → 즉시 결함 (노출 자체가 위반).
4. C등급 발견 → 한정 표지("~로 전해진다", "전기에 따르면" 등) 동반 여부 확인. 단정형 서술이면 결함 — 등급의 무단 승격이다.
5. disputed 사건이 연표·본문에서 상충 표시 없이 단일 설로 서술되면 결함.

이 검사가 중요한 이유: 작성·교열·구현의 어느 단계에서든 "문장을 매끄럽게" 다듬는 과정이 한정 표지를 지우기 쉽고, 그 결과는 미검증 주장이 검증된 사실의 얼굴로 배포되는 것이다.

## 5. 결함 보고 형식

보고서는 `_workspace/05_qa/content_qa_report.md`. 모든 결함은 **근거쌍**(사이트 텍스트 vs DB 레코드)을 포함하라 — 근거쌍이 없으면 수정자가 판정을 재현할 수 없고, 재현 불가한 결함 보고는 분쟁만 만든다.

```markdown
### 결함 CQA-007 [날짜 불일치] 심각도: 높음
- 위치: site/data/pages/life.json > sections[3].body / life.html "신민회" 절
- 사이트 텍스트: "1908년 9월 평양에 대성학교를 세웠다"
- DB 레코드: timeline.json #evt-0042 — date.start "1908-09-26", precision "day",
  place "평양" / claims_register #clm-0118 grade A
- 판정 근거: 날짜 값은 일치하나 본문 인용문과 연도 표기 검증 필요 → 9월 26일 vs 9월: precision 손실은
  허용(축약), 값 충돌 아님 — 단 동일 절의 "1907년 설립" 표기는 값 불일치로 결함
- 담당: narrative-writer (원문) / data-engineer (데이터 반영 확인)
```

- 심각도: 높음(값 오류·D등급 노출·인용 변형) / 중간(precision 위반·한정 표지 누락·고아 참조) / 낮음(표기 규칙 위반).
- 통계 요약을 보고서 서두에 표로 제시하라 — 전수성의 증명이다. 검사 건수가 비정상적으로 적으면(예: 60+개 사건 사이트에서 날짜 10건) 결함이 없는 것이 아니라 추출 단계가 누락된 것이므로, 대조보다 추출을 먼저 의심하라.

```markdown
| 검사 항목 | 추출 건수 | 대조 건수 | 결함 |
|----------|----------|----------|------|
| 날짜     | 214      | 214      | 3    |
| 인명     | 187      | 187      | 1    |
| 인용문   | 23       | 23       | 0    |
| 각주 마커 | 156      | 156      | 2    |
```

- 수정은 직접 하지 않는다 — 서브 모드이므로 보고서를 반환하고, 수정은 오케스트레이터가 해당 작성자를 재호출해 수행한다. 수정 후에는 결함 항목 재검만이 아니라 해당 페이지 전수 재대조를 하라 — 수정 과정에서 주변 텍스트가 함께 바뀌는 일이 흔하다.

## 산출물 검증

1. **전수성:** 보고서의 항목별 검사 건수가 추출 단계의 항목 수와 일치하는가 — 추출 1,200건에 대조 800건이면 400건은 검사하지 않은 것이다.
2. **근거쌍 완전성:** 모든 결함에 사이트 텍스트 원문 + DB 레코드 id/값 + 위치가 있는가.
3. **양방향성:** 각주 검사가 마커→매핑, 매핑→마커 양방향으로 수행되었는가.
4. **데이터 주도 텍스트 포함 여부:** HTML 정적 텍스트만 검사하고 data/*.json을 빠뜨리지 않았는가 — 이 사이트의 본문 대부분은 JSON에 있다.
5. **판정 일관성:** "사소해서 통과" 처리된 항목이 0건인가 — 발견되면 전수 원칙 위반으로 보고서를 재작성하라.
