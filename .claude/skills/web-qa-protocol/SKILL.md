# 웹 QA 프로토콜 (web-qa-protocol)

이 스킬은 여러 에이전트가 병렬로 만든 모듈들이 실제로 맞물려 동작하는지를 검증하는 방법을 정의한다. 핵심 전제: **각 모듈은 그럴듯한데 통합이 깨지는 것이 다중 작성자 코드베이스의 기본 실패 모드다.** 따라서 QA의 단위는 파일이 아니라 경계면(데이터 ↔ 코드, 링크 ↔ 대상, 코드 ↔ 브라우저)이며, 판정 근거는 추론이 아니라 실행과 대조다.

## 1. 경계면 교차 비교법 — 이 스킬의 핵심

### "존재 확인"은 QA가 아니다

다음은 모두 통과 사유가 **아니다**.

- "timeline.json 파일이 존재한다"
- "timeline.js 코드가 그럴듯하게 작성되어 있다"
- "각자 따로 보면 둘 다 문제없어 보인다"

이유: 통합 버그의 대부분은 각 산출물 내부가 아니라 **경계면의 불일치**에서 발생한다. 데이터를 만든 에이전트와 소비 코드를 만든 에이전트는 서로 다른 머릿속 스키마로 작업했고, 각자의 산출물은 각자의 가정 안에서는 완벽하다. JSON은 `date.start`를 내보내는데 JS는 `event.date_start`를 읽는 코드는, 파일 단위로는 둘 다 "정상"이지만 실행하면 모든 날짜가 undefined다. 이런 결함은 파일을 하나씩 보는 방식으로는 구조적으로 발견할 수 없다 — 양쪽을 동시에 열어야만 보인다.

### 교차 비교 절차

1. **데이터 쪽:** JSON 파일을 열고 실제 레코드의 필드 구조를 추출한다 (첫 레코드만 보지 말고 표본 여러 개 — 필드가 있는 레코드와 없는 레코드가 섞여 있을 수 있다).
2. **코드 쪽:** 소비 JS를 열고 데이터 객체에서 읽는 모든 참조를 추출한다 — `event.title`, `e.date.start`, `d["place"]`, 구조 분해 `const {summary} = event` 전부.
3. **필드 단위 대조:** 코드가 읽는 모든 경로가 데이터에 실제 존재하는가? 타입이 맞는가(배열을 기대하는데 객체인가)? 데이터에는 있는데 코드가 안 쓰는 필드는 의도된 것인가(계약의 소비 필드 목록과 대조)?
4. **null/누락 경로 검사:** 데이터에서 null이거나 누락 가능한 필드(계약의 null 정책 참조)에 대해 코드가 방어하는가? `place.lat`이 null인 레코드 하나가 지도 전체를 죽이는 패턴이 전형적이다.
5. **불일치는 어느 쪽이 맞는지 계약(architecture.md)으로 판정**하고, 틀린 쪽의 소유자에게 결함 통보. 계약 자체가 모호하면 web-architect에 계약 보완을 요구하라.

코드 쪽 참조 추출은 grep으로 시작할 수 있다 (정규식이 전부를 잡지는 못하므로 결과를 코드 읽기로 보완하라).

```bash
# JS에서 데이터 객체 필드 참조 후보 추출
grep -oE '(event|e|d|item|record)\.[a-zA-Z_][a-zA-Z0-9_.]*' site/js/timeline.js | sort -u
grep -oE 'const \{[^}]+\}' site/js/timeline.js   # 구조 분해 참조
```

대조 결과는 필드 대조표로 보고서에 남겨라 — 판정의 재검증이 가능해야 한다.

```markdown
| 코드 참조 (timeline.js) | 데이터 존재 (timeline.json) | 계약 | 판정 |
|---|---|---|---|
| event.date.start | 60/60 레코드 | 필수 | 통과 |
| event.dateStart  | 0/60 레코드  | 없음 | 결함 — 코드가 계약 외 필드 참조 |
| event.place.lat  | 41/60 (19건 null) | null 허용 | 코드 null 방어 확인 → 통과 |
```

## 2. Incremental QA — 모듈 완성 직후 검사

전체 완성 후 일괄 QA는 두 가지 이유로 실패한다: (1) 결함이 누적된 뒤에는 원인 모듈 추적 비용이 곱으로 늘고, (2) 마감 압력 속의 일괄 수정은 새 결함을 만든다. 그래서 QA는 모듈 완성 **직후** 단위로 수행한다.

**절차:** 개발자의 완성 고지(또는 변경 감지) → 30분 내 해당 모듈만 검사 → 통과/결함을 즉시 통보.

모듈 단위 검사 범위:
1. 해당 모듈의 경계면 교차 비교 (1절)
2. 해당 페이지를 로컬 서버로 열어 콘솔 에러 0건 확인
3. 해당 모듈이 추가한 링크·이미지 경로의 실존 확인 (check_links.py)
4. 해당 모듈이 소비하는 데이터의 스키마 검사 (check_data_schema.py)

결함 발견 시 해당 개발자에게 **재현 절차 + 기대 동작 + 실제 동작**을 함께 통보하라. "안 됨"만 전달하면 개발자가 재현에 시간을 쓰고, 그 시간은 전부 낭비다. 수정 후에는 결함 부분만이 아니라 모듈 검사를 다시 수행하라 — 수정이 새 결함을 만드는 일은 흔하다.

## 3. 최종 QA 체크리스트 (Phase 5)

전 모듈 incremental 통과 후에도 전체 통합 검사는 별도로 수행한다 — 모듈 간 상호작용은 모듈 단위 검사에 안 잡힌다.

| # | 항목 | 방법 | 통과 기준 |
|---|------|------|----------|
| 1 | 내부 링크·리소스 전수 | `check_links.py site/` | 끊어진 참조 0건 |
| 2 | 데이터 스키마 | `check_data_schema.py` (파일별) | 필수 필드 누락 0건 |
| 3 | 콘솔 에러 | 로컬 서버로 전 페이지 열어 콘솔 확인 | 에러 0건 (경고는 목록화) |
| 4 | 데이터 로드 실패 처리 | data/ 파일 1개를 임시 리네임 후 페이지 열기 | 에러 메시지 표시 (빈 화면 불가) |
| 5 | 반응형 | 360px / 768px / 1280px 3뷰포트 | 가로 스크롤·겹침 없음 |
| 6 | 이미지 alt | HTML 전수에서 `<img>` 추출 | alt 누락 0건 |
| 7 | 키보드 | 인터랙티브 컴포넌트 전수 조작 | 도달·작동 불가 0건 |
| 8 | 성능 예산 | 5절 측정법 | 예산표 전 항목 통과 |

검사는 반드시 `cd site && python3 -m http.server 8000` 환경에서 — file://의 fetch 차단은 결함이 아니다 (site-architecture 스킬 참조).

## 4. 번들 검증 스크립트

스크립트는 이 스킬의 `scripts/`에 번들되어 있다. 손으로 훑는 검사는 누락이 생기므로, 기계적으로 전수 가능한 검사는 반드시 스크립트로 하라. 두 스크립트 모두 결함 발견 시 exit code 1을 반환한다 — exit code로 게이트 판정을 자동화할 수 있다.

### check_links.py — 링크·리소스 전수 대조

site/ 내 모든 HTML에서 href/src를 추출해 로컬 파일 실존 여부를 대조한다.

```bash
python3 .claude/skills/web-qa-protocol/scripts/check_links.py site/
# 출력: 끊어진 참조 목록 (어느 HTML의 어떤 참조가 어떤 경로를 못 찾는지)
# exit 0 = 전부 통과, exit 1 = 끊어진 링크 존재
```

외부 URL(http/https)은 존재 검사 대상이 아니다(네트워크 의존 배제) — 목록만 출력하므로 별도로 표본 확인하라.

### check_data_schema.py — JSON 필수 필드·타입 검사

JSON 파일과 필수 필드 목록을 받아 모든 레코드를 전수 검사한다. 필드명은 `이름:타입` 형식으로 타입까지 검사할 수 있다(타입: str, int, float, number, bool, list, dict).

```bash
python3 .claude/skills/web-qa-protocol/scripts/check_data_schema.py \
  site/data/timeline.json --records events \
  --required id:str title:str date:dict summary:str sources:list confidence:str
# --records: 레코드 배열이 위치한 키 경로 (점 표기, 생략 시 최상위 배열로 간주)
# exit 0 = 전부 통과, exit 1 = 누락/타입 불일치 존재
```

필수 필드 목록은 추측하지 말고 architecture.md의 인터페이스 계약에서 가져와라 — 스크립트는 계약의 집행 도구이지 계약의 대체물이 아니다.

## 5. 성능 예산표와 측정법

| 항목 | 예산 | 측정법 |
|------|------|--------|
| 첫 로드 전송량 (이미지 제외) | < 200KB | 페이지별 HTML+CSS+JS+JSON 파일 크기 합산 (`du`/`wc -c`) |
| 개별 JSON | < 150KB | 초과 시 data-engineer에 분할 로딩 요청 |
| 본문 이미지 1장 | < 200KB (WebP) | 파일 크기 확인, 초과분 목록화 |
| 페이지 내 요청 수 | < 25 | HTML의 src/href + fetch 호출 수 집계 |

측정 없는 최적화 판정 금지 — "무거워 보인다"는 결함이 아니고, "가벼워 보인다"는 통과가 아니다. 예산 초과 항목은 수치와 함께 performance-optimizer 또는 해당 소유자에게 전달하고, 최적화 후 같은 방법으로 재측정해 전후 수치를 보고서에 남겨라.

## 산출물 검증

QA 보고서(`_workspace/05_qa/*_report.md`) 자체의 품질 기준이다.

1. 모든 판정에 **방법과 근거**가 있는가 — "통과"라고만 쓴 항목은 검사하지 않은 항목으로 간주한다.
2. 경계면 교차 비교 결과가 필드 대조표로 첨부되어 있는가.
3. 두 스크립트의 실행 출력(exit code 포함)이 보고서에 인용되어 있는가 — 실행하지 않은 스크립트의 결과를 적지 마라.
4. 결함 항목마다 재현 절차·기대/실제 동작·담당 소유자가 명시되어 있는가.
5. 수정 후 재검증 기록이 있는가 — 결함 통보로 끝난 QA는 절반만 한 것이다.
